## 면접 전 체크리스트

제조 현장 AI 엔지니어 면접은 일반 ML 엔지니어 면접과 결이 다릅니다. **정확도**보다 **레이턴시·안정성·운영성**을 더 깊이 묻고, 라인 정지와 불량 유출이라는 실제 비용을 이해하는지 확인합니다.

```
핵심 준비 포인트
  ├── 레이턴시 예산 계산을 숫자로 말할 수 있는가
  ├── FP16/INT8 트레이드오프를 설명할 수 있는가
  ├── SLA 위반 시 대응 프로세스를 순서대로 말할 수 있는가
  └── 모델 배포 → 모니터링 → 재학습 사이클 전체를 그릴 수 있는가
```

## 화이트보드 설계 문제 3종

### 시나리오 1: 신규 비전 검사 라인 설계

> **면접관:** "현대차 울산 공장 도장 검사 라인을 새로 구축합니다. 카메라 8대, 사이클타임 60초, Escape Rate 0.01% 목표입니다. 시스템을 어떻게 설계하겠습니까?"

**모범 답변 구조:**

**① 요구사항 명확화 (먼저 질문)**
```
"몇 가지 확인하겠습니다."
- 탐지해야 할 불량 유형 몇 가지인가요?
- 기존 MES 시스템과 연동이 필요한가요?
- 엣지 처리인지 중앙 서버 처리인지 방향이 있나요?
- 야간 조명 조건도 동일한가요?
```

**② 레이턴시 예산 산정 (숫자로 제시)**
```
사이클타임 60초 / 카메라 8대 → 카메라당 평균 간격 7.5초

동시 처리 고려 시 최대 허용 레이턴시 = 1,500ms

단계별 배분:
  카메라 캡처 + 전송   100ms
  GPU 전처리           50ms
  TRT FP16 추론       200ms   ← 핵심
  후처리 + NMS         80ms
  PLC/MES 전송         70ms
  합계                500ms   (여유 3×)
```

**③ 아키텍처 설계**
```
카메라 8대 → 4대씩 2개 GPU 서버로 분산
각 서버: RTX 4090 또는 A100
멀티 CUDA Stream으로 4대 병렬 처리
Failover: 1대 서버 장애 시 나머지가 8대 전담
```

**④ 운영 체계**
```
Prometheus + Grafana 모니터링
Escape Rate 0.01% 초과 → 즉시 알람
Shadow 배포로 안전한 모델 교체
월 1회 정기 재학습 트리거
```

> **면접 핵심:** 질문으로 요구사항을 먼저 명확화하는 자세, 레이턴시 예산을 구체적 숫자로 제시, 트레이드오프 언급, 운영까지 고려한 전체 사이클 설명.

### 시나리오 2: 성능 저하 원인 분석

> **면접관:** "월요일 출근했더니 지난 금요일 오후부터 FRR(False Reject Rate)이 2%→8%로 올랐습니다. 어떻게 분석하겠습니까?"

**5단계 체계적 진단:**

| 단계 | 행동 | 확인 사항 |
|------|------|---------|
| **1. 시간 특정** | Grafana 확인 | 급격히? 서서히? 특정 카메라만? |
| **2. 변경 이력** | 변경 관리 시스템 | 모델 업데이트, 조명 교체, SW 배포 |
| **3. 데이터 분석** | PSI, 신뢰도 분포 | 특정 불량 유형에서만? 시야각 의존? |
| **4. 임시 조치** | 임계값 일시 조정 | FRR을 허용 범위로 복구 (생산성 우선) |
| **5. 근본 해결** | 원인별 대응 | 조명→복원/재학습, 코드→롤백 |

```python
# Grafana에서 FRR 급증 시점 쿼리
sum(rate(vision_inspections_total{result="false_reject"}[5m]))
/
sum(rate(vision_inspections_total[5m]))
```

> **면접 핵심:** "느낌이 아니라 Grafana 수치로 말하기." 비즈니스 영향(생산성 손실)과 기술 원인 분석을 동시에 언급. 이번 케이스를 SOP에 추가하는 예방 관점 언급.

### 시나리오 3: 레이턴시 최적화

> **면접관:** "현재 추론 P99 레이턴시가 95ms, SLO는 50ms입니다. 어떻게 개선하겠습니까?"

**단계적 최적화 접근:**

```
1단계 — 프로파일링 (병목 찾기)
  nsys profile python infer.py
  → 전처리:20ms, 추론:60ms, 후처리:15ms
  → 추론이 최대 병목 확인

2단계 — 모델 최적화 (가장 큰 효과)
  FP32 → FP16 전환 → 추론 30ms 예상 (2× 향상)
  정확도 검증 (1,000장 검증셋, mAP ΔΔ < 0.3%)

3단계 — 파이프라인 병렬화
  전처리-추론 스레드 분리
  CUDA Stream 비동기 오버랩 → 전체 ~20% 감소

4단계 — 모델 경량화 검토 (트레이드오프)
  YOLOv8m → YOLOv8s: 추론 40% 단축
  단, Escape Rate 재검증 필수

5단계 — 검증
  Shadow 배포 24시간 → 실제 P99 측정
  FP16 전환 후 Escape Rate 7일 모니터링
```

**예상 결과:**
```
최적화 전:  P99 95ms
FP16 적용:  P99 48ms ← SLO 달성 ✓
파이프라인: P99 39ms ← 여유 확보
```

## 핵심 기술 Q&A 30선

### 추론 엔진 기초

**Q1. 학습과 추론에서 GPU 메모리 사용량이 왜 다른가요?**

> 학습 시에는 역전파를 위해 모든 레이어의 활성화값을 메모리에 저장합니다. 추론 시에는 순전파만 수행하므로 현재 레이어의 입출력만 유지하면 됩니다. 같은 모델이라도 추론 메모리는 학습의 **1/3~1/5** 수준입니다. 이것이 추론 특화 GPU(T4, A2)를 사용하는 이유입니다.

---

**Q2. `model.eval()`을 빠뜨리면 무슨 일이 생기나요?**

> BatchNormalization이 저장된 running statistics 대신 현재 배치 통계를 사용합니다. 배치 크기 1인 추론에서 BatchNorm이 불안정해져 예측 불가능한 출력이 나옵니다. Dropout도 활성화되어 매번 다른 결과가 나옵니다. 반드시 `model.eval()` + `torch.no_grad()` 쌍으로 사용해야 합니다.

---

**Q3. `torch.cuda.synchronize()`가 레이턴시 측정에 왜 필요한가요?**

> CUDA 연산은 기본적으로 **비동기**로 실행됩니다. `synchronize()` 없이 시간을 측정하면 CPU가 GPU 작업을 큐에 제출한 시간만 측정되어 실제보다 훨씬 짧게 나옵니다. 정확한 레이턴시 측정을 위해 측정 구간 앞뒤로 `torch.cuda.synchronize()`를 호출해야 합니다.

---

**Q4. TensorRT 엔진을 다른 GPU에서 사용할 수 없는 이유는?**

> TensorRT 빌드 과정에서 해당 GPU의 아키텍처(Ampere, Ada Lovelace 등)에 맞는 CUDA 커널을 선택·컴파일합니다. 이 엔진 파일은 **특정 GPU 아키텍처에 종속**됩니다. T4용 엔진을 A100에서 실행하면 오류 또는 성능 저하가 발생합니다. GPU 드라이버나 TRT 버전이 변경돼도 재빌드가 필요합니다.

---

**Q5. FP16 적용 시 오버플로우가 발생하는 경우와 해결법은?**

> FP16의 최대값은 65,504입니다. Softmax나 LayerNorm 내부의 지수 연산이 이를 초과하면 `inf`가 됩니다. 해결 방법: 해당 레이어를 FP32로 고정(`layer.precision = trt.DataType.FLOAT`), 또는 입력 정규화를 강화합니다. TensorRT는 대부분 자동 처리하지만, 커스텀 레이어에서는 직접 처리해야 합니다.

---

**Q6. INT8 캘리브레이션에서 캘리브레이션 데이터셋이 중요한 이유는?**

> INT8은 각 레이어의 활성화값 분포를 사전 측정해 스케일 인자를 결정합니다. 캘리브레이션 데이터가 실제 운영 환경과 다르면 스케일 인자가 잘못 설정되어 정확도가 급락합니다. **반드시 실제 라인 환경 이미지**로 구성해야 하며, 최소 500장~1,000장, 조명 조건과 불량/정상 다양성을 모두 포함해야 합니다.

---

### ONNX & 변환

**Q7. ONNX opset 버전을 명시해야 하는 이유는?**

> ONNX 연산자는 버전별로 정의가 다릅니다. opset을 명시하지 않으면 PyTorch 기본값이 적용되는데, 이것이 TensorRT가 지원하는 opset과 달라 변환에 실패할 수 있습니다. TRT 8.x/9.x 환경에서는 **opset 17**을 명시적으로 지정하는 것이 권장됩니다.

---

**Q8. `onnxsim`을 사용하면 TRT 변환 성공률이 높아지는 이유는?**

> 변환 직후 ONNX 그래프에는 Identity 노드, 불필요한 Transpose, 중복 Cast 등 최적화되지 않은 연산이 남아 있습니다. onnxsim은 이를 제거하고 그래프를 단순화합니다. TensorRT의 ONNX 파서가 처리해야 할 노드 수가 줄어들어 변환 성공률이 높아지고 빌드 시간도 단축됩니다.

---

**Q9. Trace vs Script 방식의 차이와 선택 기준은?**

> Trace는 dummy_input으로 한 번 실행해 해당 경로만 그래프로 기록합니다. `if/for` 분기가 고정됩니다. Script는 Python 문법을 정적으로 분석해 모든 분기를 처리합니다. **일반 CNN/Transformer는 Trace**, 학습/추론 분기가 남아 있거나 동적 제어 흐름이 있으면 **Script**를 선택합니다.

---

### MLOps & 운영

**Q10. SLO와 SLA의 차이는? 왜 SLO가 SLA보다 엄격해야 하나요?**

> SLA는 고객·운영팀과의 외부 계약 수준, SLO는 내부 달성 목표입니다. SLO가 SLA보다 엄격해야 하는 이유는 **에러 버짓(Error Budget)** 때문입니다. SLO를 SLA와 같게 설정하면 SLO가 깨지는 순간 SLA도 동시에 깨집니다. SLO를 더 엄격하게 설정해두면 SLO 위반 시 즉시 대응해도 SLA는 지킬 수 있는 버퍼가 생깁니다.

---

**Q11. Shadow 배포와 Canary 배포의 차이는?**

> Shadow 배포는 새 모델이 실제 운영 데이터로 추론하지만 결과를 적용하지 않습니다. 운영 영향 없이 새 모델을 실제 데이터로 검증합니다. Canary 배포는 실제 요청의 일부(예: 5%)를 새 모델로 처리합니다. 단계적으로 비율을 높이며 위험을 최소화합니다. 제조 현장에서는 **Shadow 24시간 → Canary 10% 4시간 → 전체 전환** 순서를 권장합니다.

---

**Q12. 데이터 드리프트를 조기에 감지하는 방법은?**

> 모델 신뢰도 점수 분포를 기준 분포와 비교하는 **KS 검정(Kolmogorov-Smirnov test)**이 실용적입니다. 신뢰도 P50이 배포 초기 대비 지속적으로 하락하면 조명 변동, 렌즈 오염, 제품 변경을 의심합니다. Prometheus의 `vision_model_confidence_score` 메트릭에 7일 이동 평균 트렌드 알림을 추가하면 급격한 스파이크보다 완만한 하락을 먼저 감지할 수 있습니다.

---

**Q13. P99 레이턴시가 P50의 3배 이상일 때 의심해야 할 원인은?**

> GPU 메모리 경쟁(다른 프로세스와 공유), CPU 전처리 GIL 병목, GC(Garbage Collection) 일시 중단, 네트워크 패킷 손실로 인한 재전송, 온도 상승으로 인한 GPU 쓰로틀링이 주요 원인입니다. `nvidia-smi dmon`으로 실시간 GPU 상태를 확인하고, `nsys`로 프로파일링해 레이턴시 스파이크 구간을 특정합니다.

---

### 비전 검사 도메인

**Q14. 자동차 비전 검사에서 FNR(미검출률)을 FPR(오검출률)보다 중요하게 보는 이유는?**

> 불량을 놓치면(FN) 완성차 리콜로 이어질 수 있어 수십억~수백억 원의 비용이 발생합니다. 정상을 불량으로 판정하면(FP) 이젝터가 더 작동하고 작업자가 수동 확인하는 수준입니다. 리스크 비대칭성이 매우 크기 때문에 FNR 최소화가 절대 원칙입니다. 임계값 튜닝 시에도 FNR이 허용 기준을 초과하지 않는 범위에서 FPR을 낮춥니다.

---

**Q15. 사이클타임이 55ms인 라인에서 P99 레이턴시 SLA를 어떻게 설정하나요?**

> 단순히 55ms를 SLA로 설정하면 위험합니다. 내부 SLO는 **45ms** (여유 10ms 확보), 외부 SLA는 **55ms**로 설정합니다. 레이턴시 예산은 각 단계(캡처→전처리→추론→후처리→PLC 전송)에 배분하고, P99가 아닌 **P99.9**까지 SLA 이내인지 검증합니다. 단 하나의 레이턴시 초과도 라인 정지로 이어질 수 있기 때문입니다.

---

**Q16. 공장에서 조명 교체 후 모델 성능이 급락했습니다. 빠른 대응 방법은?**

> ① 신뢰도 점수 분포 확인 → 드리프트 확인 ② 이전 조명 조건 이미지와 새 조명 이미지의 픽셀 분포 비교(PSI) ③ 임시 조치: 임계값 조정 ④ 새 조명 조건에서 50~200장 레이블링 → Fine-tuning 또는 Domain Adaptation ⑤ Shadow 배포로 새 모델 검증 후 교체. 장기적으로는 조명 모니터링 센서를 추가해 조명 변동을 자동 감지합니다.

---

### 시스템 설계

**Q17. 엣지(Edge) 처리와 서버 처리를 언제 선택하나요?**

| 기준 | Edge 선택 | Server 선택 |
|------|---------|-----------|
| **레이턴시** | < 50ms 요구 | 100ms+ 허용 |
| **네트워크** | 불안정 환경 | 안정적 10GbE |
| **카메라 수** | 1~4대 | 8대 이상 |
| **모델 복잡도** | 경량 (YOLOv8s) | 복잡 (Transformer) |
| **전력 제약** | 저전력 필요 | 제약 없음 |

> 자동차 라인은 대개 **하이브리드**: 엣지에서 1차 실시간 판정, 서버에서 복잡한 이상 탐지와 재학습 파이프라인.

**Q18. Modbus TCP vs PROFINET, 어떤 기준으로 선택하나요?**

> Modbus TCP는 설정이 단순하고 범용적이지만 사이클 시간이 1~10ms입니다. PROFINET RT는 사이클 시간 1~4ms, PROFINET IRT는 <1ms로 더 정밀한 동기화를 지원합니다. 카메라 트리거와 이젝터 신호처럼 타이밍이 중요한 제어 신호에는 **PROFINET RT 이상**을 권장합니다. 레거시 시스템과의 호환성이나 예산 제약이 있으면 Modbus TCP를 선택합니다.

## 면접관이 좋아하는 답변 패턴

**숫자로 말하기**
```
❌ "FP16이 더 빠릅니다."
✅ "FP16은 FP32 대비 P99 레이턴시 기준 1.8~2.5배 빠르며,
    mAP 손실은 0.1~0.3% 수준입니다. 단, FP16 최대값 65,504를
    초과하는 레이어는 FP32로 고정이 필요합니다."
```

**트레이드오프 언급하기**
```
❌ "INT8을 쓰면 됩니다."
✅ "INT8은 속도가 2~4배 빠르지만 캘리브레이션 데이터 품질에
    민감합니다. 라인 환경 이미지 1,000장으로 캘리브레이션하고,
    7일간 Escape Rate를 Shadow 배포로 검증한 뒤 적용합니다."
```

**운영까지 생각하기**
```
❌ "모델을 배포하면 됩니다."
✅ "Shadow 24시간 → Canary 10% → 전체 전환 순서로 진행합니다.
    전환 후 1시간은 온-콜 담당자가 집중 모니터링하고,
    Escape Rate가 기준 초과 시 자동 롤백 조건을 미리 설정합니다."
```
